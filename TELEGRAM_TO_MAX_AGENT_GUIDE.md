# Инструкция для LLM-агента: перенос Telegram-бота в MAX

Документ нужен для ситуации, когда разработчик дает нейросети готовый код Telegram-бота, а нейросеть должна перенести его в мессенджер MAX.

Цель: не переписать проект с нуля, а аккуратно сохранить бизнес-логику Telegram-бота и заменить только транспортный слой, интерфейсные элементы, сценарии сообщений, команды, callback-обработку и деплойные настройки под MAX.

Эта инструкция написана как рабочий регламент для LLM-агента. Агент должен идти по этапам, фиксировать результаты, не ломать уже работающие платежи, подписки, базу данных и внешние API.

---

## 0. Главный принцип миграции

### Что делает агент

Агент получает Telegram-бота и превращает его в MAX-бота.

Правильный подход:

1. Изучить проект.
2. Понять, какие части завязаны именно на Telegram.
3. Сохранить бизнес-ядро.
4. Создать MAX-слой рядом или вместо Telegram-слоя.
5. Проверить связь с MAX на простом echo-сценарии.
6. Перенести меню, кнопки, сценарии, платежи, рассылки и админку.
7. Запустить локально.
8. Выкатить на сервер.
9. Проверить живой сценарий от `/start` до оплаты и результата.

Неправильный подход:

1. Сразу переписывать весь проект.
2. Удалять старую рабочую Telegram-логику до понимания архитектуры.
3. Менять схему БД без необходимости.
4. Менять платежную бизнес-логику одновременно с переносом интерфейса.
5. Переносить Telegram-кнопки и Telegram-команды один в один, не учитывая ограничения MAX.

### Что считать ядром проекта

Ядро проекта обычно остается почти без изменений:

- `database/`
- `services/`
- платежи
- подписки
- тарифы
- внешние API генерации
- промокоды
- реферальная логика
- статистика
- рассылки как бизнес-идея
- админские операции
- Docker-деплой

### Что считать транспортным слоем

Транспортный слой почти всегда переписывается:

- Telegram handlers
- Telegram routers
- Telegram FSM
- Telegram inline keyboards
- Telegram reply keyboards
- Telegram invoices / Stars
- Telegram parse mode нюансы
- Telegram file_id
- Telegram bot commands
- Telegram deeplink format
- Telegram message deletion/editing behavior

В MAX нужно создать свой слой:

- `max_handlers/`
- `max_keyboards/`
- `services/*_max.py`, если сервис зависит от объекта `bot`
- MAX-compatible adapters для отправки сообщений, файлов, стикеров, кнопок и уведомлений

## 1. Этап аудита исходного Telegram-бота

### Задача этапа

Понять, как устроен Telegram-бот, что в нем является бизнес-логикой, а что является Telegram-обвязкой.

Агент не должен писать код до первичного аудита.

### Что обязательно посмотреть

Файлы входа:

- `main.py`
- `config.py`
- `.env`
- `.env.example`
- `requirements.txt`
- `Dockerfile`
- `docker-compose.yml`

Папки логики:

- `handlers/`
- `services/`
- `database/`
- `keyboards/`
- `media/`
- `prompts/`
- `tools/`

Если проект использует другую структуру, агент должен сначала составить карту проекта.

### Что выписать в рабочий план

Агент должен создать отдельный файл плана, например:

```text
MAX_MIGRATION_PLAN.md
```

В нем фиксируются:

- что уже изучено;
- какие файлы завязаны на Telegram;
- какие файлы можно оставить почти без изменений;
- какие риски найдены;
- какие этапы уже выполнены;
- что осталось проверить;
- какие команды запускались;
- какие ошибки были исправлены.

### Минимальная карта Telegram-зависимостей

Агент должен найти и перечислить:

- импорты `aiogram`, `telegram`, `telebot` или другой Telegram-библиотеки;
- обработчики `/start`, `/help`, `/balance`, `/admin`;
- callback handlers;
- FSM states;
- inline keyboards;
- reply keyboards;
- оплату Telegram Stars;
- работу с Telegram `file_id`;
- отправку фото, документов, видео, стикеров;
- `edit_message_text`;
- `delete_message`;
- `answer_callback_query`;
- `set_my_commands`;
- deeplink payload в `/start`.

Пример поиска:

```bash
rg "aiogram|telegram|telebot|CallbackQuery|Message|FSM|InlineKeyboard|ReplyKeyboard|file_id|send_photo|send_document|send_sticker|answer_callback|set_my_commands"
```

### Что нельзя делать на этом этапе

- Нельзя удалять старый Telegram-код без бэкапа.
- Нельзя менять `.env` без понимания, какие переменные реально используются.
- Нельзя менять базу данных до проверки `database/db.py` и `database/crud.py`.
- Нельзя менять тарифы, цены, лимиты и платежи без отдельной команды пользователя.
- Нельзя одновременно менять продуктовую логику и транспортный слой, если это не требуется.

### Результат этапа

В конце этапа агент должен записать в план:

```text
Этап 1 завершен.
Изучены: main.py, config.py, handlers, services, database, keyboards.
Telegram-зависимости найдены в: ...
Бизнес-ядро для сохранения: ...
MAX-слой для создания: ...
Риски: ...
```

## 3. Этап первичной проверки MAX API

### Задача этапа

Перед переносом всего проекта убедиться, что токен MAX-бота работает.

Агент должен сначала сделать простой echo-тест, а не переносить весь бот вслепую.

### Переменные окружения

В `.env` добавить:

```env
MAX_BOT_TOKEN=
```

Если проект временно поддерживает старый Telegram-токен:

```env
BOT_TOKEN=
MAX_BOT_TOKEN=
```

Приоритет должен быть у `MAX_BOT_TOKEN`.

### Минимальный echo-скрипт

Файл:

```text
tools/max_echo_test.py
```

Принцип:

1. загрузить `.env`;
2. создать `Bot`;
3. создать `Dispatcher`;
4. зарегистрировать обработчик входящих сообщений;
5. отвечать тем же текстом;
6. запустить polling.

Псевдокод:

```python
from maxapi import Bot, Dispatcher, Router
from maxapi.enums.parse_mode import ParseMode
from dotenv import load_dotenv
import os

load_dotenv()

bot = Bot(token=os.getenv("MAX_BOT_TOKEN"), format=ParseMode.HTML)
dp = Dispatcher()
router = Router("echo")

@router.message_created()
async def echo(event):
    text = event.message.body.text or ""
    await bot.send_message(chat_id=event.chat_id, text=f"Эхо: {text}")

dp.include_router(router)
dp.start_polling(bot)
```

Реальный код зависит от версии `maxapi`, поэтому агент должен смотреть примеры библиотеки и текущий проект.

### Что проверить руками

Пользователь должен написать боту в MAX:

```text
привет
```

Ожидаемый ответ:

```text
Эхо: привет
```

### Типовые ошибки

Ошибка: бот не отвечает.

Проверить:

- верный ли токен;
- не запущен ли старый процесс с тем же токеном;
- нет ли webhook, мешающего polling;
- есть ли доступ к интернету;
- установлен ли `maxapi`;
- не падает ли импорт.

Ошибка: приходит несколько ответов.

Причина почти всегда:

- запущено несколько экземпляров бота с одним токеном;
- локальный бот и серверный бот работают одновременно;
- старый Docker-контейнер не остановлен.

Проверка:

```bash
docker compose ps
docker ps
ps aux | grep python
```

### Результат этапа

В плане фиксируется:

```text
Этап 3 завершен.
MAX-токен проверен echo-скриптом.
Polling работает.
Дубли ответов отсутствуют / найдены и устранены.
```

## 4. Этап создания MAX entrypoint

### Задача этапа

Сделать основной `main.py`, который запускает MAX-бота, подключает БД, регистрирует MAX-роутеры и фоновые задачи.

### Базовый принцип

Telegram entrypoint нельзя переносить один в один.

Нужно заменить:

- Telegram `Bot`;
- Telegram `Dispatcher`;
- Telegram router registration;
- Telegram parse mode;
- Telegram command setup;
- Telegram polling;
- Telegram session closing.

На MAX:

- `Bot` из `maxapi`;
- `Dispatcher` из `maxapi`;
- `ParseMode.HTML`;
- `max_handlers.all_routers`;
- `bot.set_my_commands`, если библиотека поддерживает;
- `dp.start_polling(bot, skip_updates=True)`;
- `bot.close_session()`.

### Рекомендуемый `main.py`

Принцип:

```python
import asyncio
import logging

from maxapi import Bot, Dispatcher
from maxapi.enums.parse_mode import ParseMode
from maxapi.types import BotCommand

from config import load_config
from database.db import setup as setup_db
from max_handlers import all_routers
from services.smart_mailer_max import smart_mailing_loop
from services.subscription_tasks_max import subscription_watcher


async def main() -> None:
    config = load_config()
    token = config.max_bot_token or config.bot_token
    if not token:
        raise RuntimeError("MAX_BOT_TOKEN is empty")

    logging.basicConfig(...)

    await setup_db(config.database_path)

    bot = Bot(token=token, format=ParseMode.HTML)
    dp = Dispatcher()
    dp.include_routers(*all_routers)

    commands = [
        BotCommand(name="start", description="Запуск бота"),
        BotCommand(name="menu", description="Главное меню"),
        BotCommand(name="balance", description="Баланс и подписка"),
        BotCommand(name="help", description="Помощь"),
    ]

    try:
        await bot.set_my_commands(*commands)
    except Exception as e:
        logger.warning("Could not set commands: %s", e)

    try:
        await bot.delete_webhook()
    except Exception:
        pass

    asyncio.create_task(subscription_watcher(bot))
    asyncio.create_task(smart_mailing_loop(bot))

    try:
        await dp.start_polling(bot, skip_updates=True)
    finally:
        await bot.close_session()


if __name__ == "__main__":
    asyncio.run(main())
```

### Важная особенность команд MAX

В некоторых версиях `maxapi` метод установки команд может работать, но выводить предупреждение, что метод устарел или отсутствует в официальной swagger-спецификации.

Это не всегда критическая ошибка.

Агент должен:

- не падать из-за ошибки установки команд;
- логировать warning;
- не блокировать запуск бота;
- понимать, что команды в UI MAX могут вести себя не так, как в Telegram.

### Регистрация роутеров

Файл:

```text
max_handlers/__init__.py
```

Пример:

```python
from .admin import router as admin_router
from .payments import router as payments_router
from .start import router as start_router
from .tarot import router as tarot_router

all_routers = (
    start_router,
    tarot_router,
    payments_router,
    admin_router,
)
```

Порядок важен.

Обычно:

1. `start`;
2. продуктовые сценарии;
3. платежи;
4. админка;
5. fallback, если он есть.

### Результат этапа

В плане фиксируется:

```text
Этап 4 завершен.
main.py переведен на maxapi.
БД инициализируется.
MAX routers подключены.
Фоновые задачи запускаются.
Команды устанавливаются с безопасным fallback.
```

## 5. Этап интерфейса и дизайна MAX-бота

### Главная задача этапа

MAX-бот должен выглядеть как самостоятельный продукт, а не как механическая копия Telegram-бота.

Агент должен особенно внимательно перенести:

- главное меню;
- кнопки;
- сценарии оплаты;
- карточку баланса;
- инструкции пользователю;
- состояние ожидания генерации;
- сообщения об ошибках;
- админские сообщения;
- рассылки.

### Базовый UX-принцип

Пользователь в MAX должен всегда понимать:

- где он находится;
- что бот ждет от него сейчас;
- что произойдет после нажатия кнопки;
- сколько у него осталось лимитов;
- что делать при ошибке;
- как вернуться в главное меню.

### Inline-кнопки

В MAX следует использовать inline-кнопки через `CallbackButton` и `LinkButton`.

Пример:

```python
from maxapi.types import CallbackButton, LinkButton
from maxapi.utils.inline_keyboard import InlineKeyboardBuilder


def main_menu_attachments():
    kb = InlineKeyboardBuilder()
    kb.row(CallbackButton(text="Задать вопрос", payload="menu:ask"))
    kb.row(CallbackButton(text="Баланс / Подписка", payload="menu:balance"))
    kb.row(CallbackButton(text="Помощь", payload="menu:help"))
    return [kb.as_markup()]
```

### Правила текста кнопок

Кнопки должны быть короткими.

Хорошо:

```text
Задать вопрос
Баланс / Подписка
Главное меню
Выбрать подписку
Оплатить
Отключить подписку
Пригласить друга
```

Плохо:

```text
Нажмите сюда, чтобы перейти в раздел, где можно задать вопрос
Купить платную подписку с автоматическим продлением
Вернуться обратно на самое главное меню
```

### Payload callback-кнопок

Payload должен быть коротким и системным.

Рекомендуемый стиль:

```text
menu:main
menu:ask
menu:balance
menu:help
menu:invite
sub:choose
sub:choose:yoo:week
sub:choose:yoo:month
sub:cancel
sub:renew_choose
tarot:open_full
admin:help
```

Нельзя хранить в callback payload длинные тексты, промпты, JSON на несколько килобайт или пользовательские сообщения.

Если нужно хранить состояние:

- использовать FSM;
- использовать `pending_actions`;
- использовать таблицу БД;
- использовать in-memory context только для краткоживущего состояния.

### Главное меню

Главное меню должно быть коротким.

Для Таро-бота пример:

```text
Выберите раздел ниже
```

Кнопки:

```text
Задать вопрос
Баланс / Подписка
Помощь
Пригласить друга
```

Если продукт сложный, допустимо 5-7 кнопок, но не больше без необходимости.

### Поведение сообщений

Рекомендуемая модель:

- основные переходы отправляют новое сообщение;
- пагинация может редактировать старое сообщение;
- технический прогресс можно удалять после завершения операции;
- ошибки лучше отправлять отдельным сообщением;
- в каждом вторичном разделе должна быть кнопка `Главное меню`.

Почему не редактировать все подряд:

- в MAX поведение редактирования может отличаться от Telegram;
- пользователю проще видеть историю действий;
- меньше риск сломать интерфейс при старых callback-событиях.

### Форматирование текста

Использовать HTML parse mode, если проект настроен на `ParseMode.HTML`.

Разрешенные элементы:

```html
<b>жирный</b>
<i>курсив</i>
<code>команда</code>
```

Нельзя без проверки использовать сложный HTML:

```html
<blockquote>
<table>
<ul>
<h1>
```

MAX может поддерживать не все, что поддерживает Telegram.

### Тон пользовательских сообщений

Текст должен быть:

- короткий;
- спокойный;
- без перегруза;
- с понятным следующим действием;
- без технических подробностей.

Пример хорошего сообщения ожидания:

```text
Выполняю расклад, смотрю ваши карты...
```

Пример плохого сообщения:

```text
Ваш запрос поставлен в очередь, сейчас выполняется обращение к внешнему API, ожидайте завершения HTTP-запроса.
```

### Состояние ожидания генерации

Если в Telegram-боте был стикер или анимация ожидания, в MAX нужно сделать аналог аккуратно.

Рабочая схема:

1. Пользователь отправляет вопрос.
2. Бот отправляет стикер ожидания, если есть `TAROT_PROGRESS_STICKER_CODE` или `TAROT_PROGRESS_STICKER_URL`.
3. Бот отправляет одно короткое текстовое сообщение ожидания.
4. Бот делает запрос к нейросети.
5. После ответа бот удаляет стикер и текст ожидания, если MAX API это позволяет.
6. Бот отправляет финальный результат.

Переменные:

```env
TAROT_PROGRESS_STICKER_CODE=
TAROT_PROGRESS_STICKER_URL=
TAROT_PROGRESS_TEXT=Выполняю расклад, смотрю ваши карты...
```

Важно:

- не отправлять много сообщений ожидания;
- не дублировать прогресс;
- не оставлять мусорные сообщения после генерации;
- если удаление не поддерживается, не падать, а просто продолжить сценарий.

### Карточка баланса

Баланс должен показывать:

- активна ли подписка;
- тариф;
- остаток лимита;
- дату обновления;
- доступные тарифы, если подписки нет;
- кнопку выбора подписки;
- кнопку главного меню.

Пример:

```text
Подписка активна
Тариф: 199 ₽ / неделя — 15 раскладов
Остаток раскладов: 14
Обновление раскладов: 2026-04-13
```

Для неактивной подписки:

```text
Подписка не активна
Расклады: 0

Подписка с автосписанием
199 ₽ / неделя — 15 раскладов
499 ₽ / месяц — 100 раскладов
```

### Оплата

Экран оплаты должен быть прямым.

Пример:

```text
Оплата через ЮKassa.
Нажмите кнопку ниже и завершите оплату.
```

Кнопки:

```text
Оплатить
Главное меню
```

После успешной оплаты обязательно отправить отдельное сообщение:

```text
Оплата прошла успешно.
Подписка активирована, расклады начислены.
```

Если оплата была внутри сценария генерации, после этого нужно автоматически продолжить сценарий через `pending_actions`.

### Админские сообщения

Админские сообщения должны быть короткими и полезными.

Они должны отвечать на вопросы:

- что произошло;
- с каким пользователем;
- какой тариф;
- какой статус;
- сколько отправлено;
- сколько ошибок;
- что делать, если есть проблема.

Пример старта рассылки:

```text
Рассылка началась
Целевая аудитория: 784 чел.
```

Пример превью рассылки:

```text
Внимание! Через 30 минут начнется автоматическая рассылка.
Текст: Сделаем прогноз по вашему вопросу и найдем практичный ориентир.
```

Пример завершения рассылки:

```text

Успешно доставлено: 99
Не доставлено (бот заблокирован): 217
Следующая рассылка через 12 часов.
```

### Рассылочная кнопка

В push-рассылке обязательно должна быть inline-кнопка с прямым действием.

Пример:

```text

```

Payload:

```text

```

Дополнительно можно оставить:

```text
Главное меню
```

Payload:


Проверить вручную:

1. `/start`

5. сообщение ожидания

11. автоматическое продолжение сценария после оплаты
12. баланс
13. помощь
14. приглашение друга
15. админ-команды
16. рассылка

### Частые UX-ошибки при переносе

Ошибка: бот отвечает несколькими разными сообщениями на одну кнопку.

Причина:

- несколько инстансов бота;
- пересекающиеся handlers;
- один callback ловится несколькими роутерами;
- нет точной фильтрации payload.

Ошибка: пользователь нажал кнопку, но ничего не произошло.

Причина:

- payload не зарегистрирован;
- callback handler ожидает другой формат;
- исключение проглочено без лога;
- бот пытается редактировать сообщение, которое нельзя редактировать.

Ошибка: админ не получает уведомления.

Причина:

- ID есть в `ADMIN_IDS`, но нет в `ADMIN_NOTIFY_IDS`;
- админ добавлен в таблицу `admins`, но рассылка читает только `.env`;
- бот не может отправить сообщение пользователю, пока тот не писал боту;
- уведомление отправляется в `chat_id`, а нужен `user_id`, или наоборот.

Решение:

- собрать notify IDs из `.env` и таблицы `admins`;
- отправлять с fallback `chat_id -> user_id`;
- логировать ошибки отправки.

В плане фиксируется:

```text

Созданы max_keyboards.
Главное меню перенесено.
Callback payload нормализованы.
Баланс и подписка отображаются корректно.
Прогресс генерации реализован одним стикером и одним текстом.
Админские сообщения приведены к формату MAX.
```

## 6. Этап переноса `/start`, deeplink, ref, promo и UTM

### Цель этапа

Перенести стартовую точку Telegram-бота в MAX так, чтобы:

1. пользователь получал правильное первое сообщение;
2. бот создавал пользователя в БД;
3. referral-ссылки продолжали работать;
4. UTM-метки не терялись;
5. промокоды из стартовой ссылки применялись;
6. повторный `/start` не ломал уже сохраненную атрибуцию;
7. приветствие выглядело естественно для MAX, а не как механически перенесенный Telegram-текст.

В Telegram стартовая ссылка обычно выглядит так:

```text
https://t.me/<bot_username>?start=ref_123
https://t.me/<bot_username>?start=promo_TEST
https://t.me/<bot_username>?start=ads_april
```

В MAX формат ссылки может отличаться. Агент не должен угадывать его по памяти. Нужно проверить фактический формат, который поддерживает текущая MAX-библиотека или официальный API.

Для MAX в проекте нужно зафиксировать:

```text
MAX bot id:
MAX username:
Формат ссылки без payload:
Формат ссылки с payload:
Как payload приходит в событии:
Какие ограничения по длине payload:
```

### Что нужно найти в Telegram-боте

Агент ищет:

```text
handlers/start.py
handlers/referral.py
handlers/promocodes.py
keyboards/main_menu.py
database/crud.py
services/notify.py
```

И все места:

```text
/start
CommandStart
get_args
payload
deep_link
ref_
promo_
utm
utm_source
referrer_id
add_user
```

Нужно понять:

1. где создается пользователь;
2. где сохраняется `utm_source`;
3. где сохраняется `referrer_id`;
4. где применяется промокод;
5. где отправляется уведомление админу о новом пользователе;
6. где показывается первое меню;
7. есть ли отдельный onboarding после `/start`;
8. есть ли логика "первый запуск" и "повторный запуск".

### Правило сохранения атрибуции

При повторном `/start` нельзя перетирать старую рекламную метку и старого реферера.

Правильная логика:

```text
Если пользователь новый:
  создать пользователя
  сохранить payload
  сохранить referrer_id или utm_source
  отправить админу "новый пользователь"

Если пользователь уже есть:
  не перетирать старые referrer_id и utm_source
  можно обновить username/first_name, если такие поля есть
  показать главное меню
```

Пример правила для БД:

```sql
utm_source = COALESCE(existing_utm_source, new_utm_source)
referrer_id = COALESCE(existing_referrer_id, new_referrer_id)
```

Самореферал запрещен:

```text
если referrer_id == user_id:
  referrer_id не сохранять
```

### Разбор payload

Стандартная логика:

```text
payload начинается с "ref_":
  это приглашение от другого пользователя

payload начинается с "promo_":
  это промокод

payload не пустой и не подходит под первые два варианта:
  это UTM-метка

payload пустой:
  обычный старт
```

Пример нормализации:

```text
raw payload: "ref_4533898"
type: referral
value: 4533898
```

```text
raw payload: "promo_FREE10"
type: promo
value: FREE10
```

```text
raw payload: "vk_ads_april"
type: utm
value: vk_ads_april
```

### MAX-реализация `/start`

В MAX не нужно слепо копировать Telegram decorators.

Плохо:

```python
@router.message(CommandStart())
async def start(message: Message):
    ...
```

Хорошо:

```python
@router.message()
async def cmd_start(event):
    text = extract_text(event)
    if not is_start_command(text):
        return
    payload = extract_start_payload(text, event)
    await process_start(event, payload)
```

Если библиотека MAX поддерживает отдельный фильтр команд, его можно использовать. Но в инструкции для агента важно правило: обработчик `/start` должен быть точным, чтобы обычное сообщение пользователя не попадало в стартовую логику.

### Извлечение user_id и chat_id

В Telegram часто:

```text
message.from_user.id
message.chat.id
```

В MAX событие может иметь другую структуру.

Агент должен сделать helper:

```python
def get_event_user_id(event) -> int | None:
    ...

def get_event_chat_id(event) -> int | None:
    ...

def get_event_text(event) -> str:
    ...
```

Эти helper-функции должны использоваться во всех обработчиках.

Нельзя в каждом handler писать разные варианты доступа к ID. Это потом приводит к ошибкам:

```text
админка проверяет один ID,
баланс пишет в другой ID,
рассылка отправляет в третий ID,
платеж привязывается не к тому пользователю.
```

### Приветствие в MAX

Приветствие не должно быть длинным. В MAX пользователь часто видит бота как обычный чат, поэтому первый экран должен быстро объяснять:

1. что делает бот;
2. что нужно нажать;
3. что будет бесплатно или платно;
4. где открыть меню.

Шаблон для Таро-бота:

```html
<b>Здравствуйте. Я сделаю расклад на 3 карты и дам подробный разбор.</b>

Задайте вопрос — я помогу увидеть направление, препятствия и возможные подсказки.

Выберите раздел ниже 👇
```

Шаблон универсальный:

```html
<b><PRODUCT_NAME></b>

Я помогу создать <PRODUCT_UNIT> по вашему запросу.

Выберите действие ниже 👇
```

Не надо писать:

```text
Добро пожаловать в нашего супер-бота, самого лучшего помощника...
```

Пользователь должен сразу понять действие.

### Главное меню после `/start`

После `/start` бот должен отправить:

1. короткое приветствие;
2. inline-клавиатуру главного меню.

Не нужно отдельно отправлять 3-4 сообщения подряд.

Плохо:

```text
Сообщение 1: привет
Сообщение 2: описание
Сообщение 3: инструкция
Сообщение 4: меню
```

Хорошо:

```text
Одно сообщение:
короткий текст + кнопки
```

### Повторный `/start`

Повторный `/start` должен работать как возврат в главное меню.

Текст:

```html
<b>Главное меню</b>

Выберите раздел ниже 👇
```

Если пользователь пришел по referral-ссылке второй раз, реферер не меняется.

Если пользователь пришел по UTM-ссылке второй раз, UTM не меняется.

Если пользователь пришел по promo-ссылке второй раз, промокод можно попытаться применить только если он еще не использован.

### Уведомление админам о новом пользователе

При первом запуске нужно отправить админам:

```html
👤 <b>Новый пользователь</b>
ID: <code>5462936</code>
Имя: Дмитрий
Метка: <code>vk_ads_april</code>
Реферер: <code>4533898</code>
```

Если данных нет:

```html
Метка: <code>нет</code>
Реферер: <code>нет</code>
```

Важно: уведомление отправляется только при первом создании пользователя, а не при каждом `/start`.

### Кнопка "Пригласить друга"

Если в Telegram была кнопка приглашения, в MAX она должна:

1. показывать персональную ссылку пользователя;
2. объяснять бонус простым текстом;
3. иметь кнопку "Главное меню".

Шаблон:

```html
🤝 <b>Пригласить друга</b>

Отправьте другу персональную ссылку.
Бонус: <b>20 раскладов</b> после первой покупки друга.

Ваша ссылка:
<code>https://max.ru/...</code>
```

Если MAX делает preview ссылки, это нормально. Не нужно бороться с preview, если он не ломает UX.

### Чек-лист этапа

```text
[ ] /start работает без payload
[ ] /start ref_<id> сохраняет referrer_id
[ ] /start promo_<code> применяет промокод
[ ] /start <utm> сохраняет utm_source
[ ] повторный /start не перетирает старую атрибуцию
[ ] самореферал игнорируется
[ ] новый пользователь пишется в БД
[ ] админ получает уведомление только один раз
[ ] главное меню открывается после /start
[ ] кнопка "Пригласить друга" генерирует MAX-ссылку
```

### Результат этапа

В план миграции агент записывает:


## 7. Этап переноса продуктового сценария генерации

### Цель этапа

Перенести главный пользовательский сценарий, ради которого существует бот.

Для Таро-бота это:

```text
пользователь выбирает раздел
пользователь задает вопрос

бот показывает пробную часть или запускает полную генерацию
бот обращается к нейросети
бот отправляет результат
```

Для другого продукта это может быть:

```text
пользователь пишет prompt
бот списывает кредиты
бот вызывает API генерации
бот отправляет файл
```

Главная задача агента: не потерять бизнес-логику, но заменить Telegram-интерфейс на MAX-интерфейс.

```text
services/openai*.py
services/kie*.py
services/replicate*.py
prompts/

FSM
pending_actions
```

Нужно составить карту сценария:

```text
3. какие состояния FSM используются

```

### Принцип переноса FSM

Если в Telegram использовался FSM, в MAX нужно сохранить саму идею состояний.

Нельзя заменять многошаговый сценарий хаотичными проверками текста.

Плохо:

```text
если пользователь написал любое сообщение:

```

Это ломает:

1. админ-команды;
2. меню;
3. оплату;
4. рассылку;
5. callback-кнопки;
6. обычные сообщения пользователя.

Хорошо:

```text
если state == waiting_question:
  сообщение считается вопросом
иначе:
  показать fallback или главное меню
```

### Основной дизайн пользовательской воронки

В MAX пользователь не должен думать, что делать дальше.

Каждый экран должен отвечать на один вопрос:

```text

```

Правильный порядок:

```text

3. Короткая инструкция
4. Пользовательский ввод
5. Прогресс-сообщение

7. Кнопки после результата
```

```text
💳 Баланс
🏠 Главное меню
```

### Экран ожидания вопроса или prompt

Текст должен быть короткий.

Для Таро:

```html
<b>Задайте вопрос Таро</b>

Примеры:
• <i>Когда в моей жизни появятся серьезные отношения?</i>
• <i>Что поможет мне увеличить доход?</i>

```

Кнопка:

```text

```

Для генератора изображений:

```html

Напишите, что должно быть на картинке, в каком стиле и с каким настроением.

Пример:
<i>Девушка у окна в стиле винтажной фотографии, теплый свет, осень.</i>
```

### Валидация пользовательского ввода

Агент должен перенести или создать валидацию:

```text
пустой текст запрещен
слишком короткий текст запрещен
слишком длинный текст обрезать или попросить сократить
команды не считать prompt
кнопочные payload не считать prompt
```

Пример:

```text
если вопрос короче 5 символов:
  "Напишите вопрос чуть подробнее."
```

```text
если вопрос длиннее 1000 символов:
  "Слишком длинный вопрос. Сократите его до 1000 символов."
```

### Проверка баланса

Перед генерацией нужно понять:

1. есть ли активная подписка;
2. есть ли остаток генераций;
3. сколько стоит действие;
4. нужно ли сохранять pending_action;
5. можно ли показать пробную часть бесплатно.

Типовой сценарий:

```text
если balance >= cost:
  списать cost
  запустить генерацию

если balance < cost:
  сохранить pending_action
  показать экран оплаты
```

Если в продукте есть пробный фрагмент:

```text
1. показать бесплатную часть
2. предложить оплату
3. после оплаты автоматически продолжить полный результат
```

### Pending action

`pending_actions` нужен, чтобы пользователь не потерял введенный вопрос после оплаты.

Пример payload:

```json
{
  "type": "tarot_question",
  "question": "Что поможет мне увеличить доход?",
  "spread": "three_cards"
}
```

После успешной оплаты:

```text
1. найти активный pending_action пользователя
2. отправить сообщение "Оплата прошла успешно"
3. продолжить сценарий
4. деактивировать pending_action
```

Важное UX-правило:

Пользователь должен увидеть отдельное сообщение об успешной оплате до продолжения сценария.

Шаблон:



Важно: если пользователь добавлен в админы, но бот никогда не переписывался с ним, MAX может не дать отправить ему уведомление. Это нужно логировать, а не считать ошибкой всей операции.

### Команда помощи для админа

В MAX команды могут быть не так удобны, как в Telegram. Поэтому нужна отдельная команда:

```text
/admin_help
```

Текст должен быть копируемым. В MAX не всегда есть Telegram-поведение "нажал на команду и она вставилась", поэтому команды нужно писать в `<code>`.

Шаблон:

```html

<code>/sub_on 5462936 15</code> — включить подписку и начислить 15 раскладов
<code>/sub_off 5462936</code> — отключить подписку
<code>/sub_check 5462936</code> — проверить подписку
<code>/sub_cancel 5462936</code> — отменить подписку
<code>/adstats vk_ads</code> — статистика по метке
<code>/adstats_all</code> — статистика по всем меткам
<code>/genpromo 10</code> — создать промокод на 10 раскладов
<code>/admin_add 123</code> — добавить админа
<code>/admin_del 123</code> — удалить админа
<code>/admin_list</code> — список админов
```

Дополнительно можно добавить inline-кнопки:

```text
📊 Статистика
👥 Админы
🎁 Промокод
🏠 Главное меню
```

Но текстовые команды оставить обязательно, потому что они быстрее для ручной работы.

Если MAX API поддерживает установку команд, агент должен:


Если MAX не поддерживает scope команд как Telegram, нужно:

```text

3. не показывать админские кнопки обычным пользователям
```

## 11. Этап Smart-рассылки и push-шаблонов

### Цель этапа

Перенести фоновую рассылку из Telegram-бота в MAX так, чтобы она:

1. запускалась автоматически;
2. отправляла сообщения нужной аудитории;
3. не спамила пользователя;
4. уведомляла админов до старта и после завершения;
5. имела понятный текст и кнопку действия;
6. не использовала лишние API-запросы к нейросети, если текст можно заранее подготовить.

Рассылка — это не просто технический цикл. Это часть интерфейса продукта. Сообщение должно выглядеть как аккуратное приглашение вернуться в бота, а не как рекламный мусор.

### Что нужно найти в Telegram-боте

Искать:

```text
services/smart_mailer.py
services/mailer.py
services/notify.py
database/crud.py
mailer_state
effects
last_effect_id
mailing
push
broadcast
```

Нужно понять:

1. как часто запускается рассылка;
2. кому она отправляется;
3. какой текст используется;
4. есть ли preview админам;
5. есть ли progress-сообщение;
6. как считаются успешные и неуспешные доставки;
7. где хранится состояние ротации;
8. как бот реагирует на блокировку пользователем.

### Аудитория рассылки

По умолчанию для SaaS-бота с подпиской правильная аудитория:

```text
пользователи без активной подписки
```

Активная подписка означает:

```text
status == "active"
current_period_end > now
```

Если продуктовая логика требует рассылать всем, это должно быть явно записано в плане миграции. Нельзя случайно поменять аудиторию с "без подписки" на "всем".

В плане нужно фиксировать:

```text
Аудитория рассылки: пользователи без активной подписки.
Периодичность: каждые 12 часов.
Исключения: заблокировавшие бота, удаленные пользователи, технические ошибки отправки.
```

### Периодичность

Базовый режим:

```text
каждые 12 часов
```

Перед отправкой:

```text
за 30 минут отправить preview админам
```

На старте:

```text
отправить админам стартовое сообщение
```

После завершения:

```text
отправить админам итоговую статистику
```

### Push-шаблоны

Если пользователь просит "пусть ИИ составит push", агент не обязан каждый раз вызывать нейросеть. Правильнее один раз подготовить набор сообщений и крутить их по кругу.

Рекомендуемый файл:

```text
prompts/mailer_push_templates.py
```

Пример структуры:

```python
PUSH_TEMPLATES = [
    "Кажется, сегодня хороший момент задать картам вопрос. Хотите посмотреть, что они подскажут?",
    "Иногда один расклад помогает увидеть ситуацию спокойнее. Задайте вопрос, если хотите разобраться.",
    "Что сейчас важнее всего понять: любовь, работа или деньги? Карты помогут собрать картину.",
]
```

Для рабочего проекта лучше иметь минимум 20 сообщений.

Требования к текстам:

```text
короткие
без давления
без обещаний гарантированного результата
без мистического перегруза
без капса
с одним понятным действием
```

Плохо:

```text
СРОЧНО! Карты уже ждут! Не упусти судьбу!
```

Хорошо:

```text
Если есть вопрос, который не отпускает, можно сделать расклад и посмотреть на ситуацию со стороны.
```

### Ротация шаблонов

Нужно хранить индекс последнего отправленного шаблона.

Варианты:

```text
mailer_state.last_template_index
или отдельное поле в mailer_state
или отдельный json/state-файл, если БД не меняется
```

Лучше использовать БД.

Логика:

```text
1. прочитать last_template_index
2. взять следующий шаблон
3. если дошли до конца списка, начать с 0
4. сохранить новый индекс
5. отправить выбранный текст
```

### Кнопка в рассылке

К каждому push-сообщению нужно добавлять inline-кнопку.

Для Таро:

```text
🔮 Задать вопрос
```

Payload:

```text
tarot:ask
```

Универсальный вариант:

```text
✨ Создать сейчас
```

Payload:

```text
product:start
```

Кнопка должна вести не просто в главное меню, а сразу в сценарий, ради которого отправляется push.

Плохо:

```text
push -> Главное меню -> пользователь думает, что делать
```

Хорошо:

```text
push -> "Задать вопрос" -> экран ввода вопроса
```

### Preview админам

За 30 минут до рассылки админам отправляется:

```html
⚠️ <b>Внимание!</b> Через 30 минут начнется автоматическая рассылка.

Текст:

Аудитория: <b>пользователи без активной подписки</b>
Кнопка: <b>🔮 Задать вопрос</b>
```

Если аудитория считается заранее, можно добавить:

```html
Примерный размер аудитории: <b>217</b>
```

### Стартовое сообщение админам

На старте:

```html

<i>Если есть вопрос, который не отпускает, можно сделать расклад и посмотреть на ситуацию со стороны.</i>
```

### Progress для админов

Если рассылка большая, раз в минуту нужно обновлять или отправлять progress:

```html
⏳ <b>Идет рассылка...</b>
```

Рекомендуемое правило:

```text

  обновлять одно сообщение
иначе:
  отправлять progress не чаще одного раза в 5 минут
```

Шаблон:


### Обработка блокировок

Если MAX возвращает ошибку, что пользователь заблокировал бота:

```text

2. не падать всей рассылкой

4. продолжить отправку следующим пользователям
```

Если отдельного поля нет, минимум логировать:

```text

```

### Темп отправки

Не нужно отправлять максимально быстро.

Базовый безопасный темп:

```text

```

```text
Запускать только один экземпляр бота.
```

## 12. Этап логов, ошибок и диагностики дублей

### Цель этапа

Сделать так, чтобы MAX-бот было понятно диагностировать.

При переносе Telegram-бота в MAX часто появляются ошибки:

```text
одна кнопка вызывает несколько ответов
бот молчит после нажатия
бот отвечает не тому пользователю
бот не видит админа
бот запускает генерацию дважды
на сервере работает старая версия
локально работает одна версия, в Docker другая
```

Этот этап нужен, чтобы агент заранее встроил нормальные логи и проверки.

### Базовый формат логов

Рекомендуемый формат:

```text
2026-04-23 18:30:15 | INFO | dispatcher | Обработано: router_id=tarot update=MESSAGE_CREATED chat_id=254066554 user_id=5462936
```

Для ошибок:

```text
2026-04-23 18:30:15 | ERROR | dispatcher | Ошибка: router_id=tarot update=MESSAGE_CREATED chat_id=254066554 user_id=5462936 cause=...
```

В логах обязательно должны быть:

```text
router_id
update_type
chat_id
user_id
callback payload
text command
state
```

Не нужно логировать:

```text
полные токены
секреты .env
секретные ключи YooKassa
полные платежные данные
личные сообщения пользователя без необходимости
```

### Логирование входящих событий

На время разработки можно включить debug-лог:

```text
event type
event id
chat id
user id
text
payload
```

Но в production нельзя писать огромный raw event на каждый update, если там есть персональные данные.

Правильный режим:

```text
development:
  подробные debug logs

production:
  короткие structured logs
```

### Диагностика дублей сообщений

Симптом:

```text
пользователь нажал одну кнопку,
а бот прислал сразу несколько разных сообщений
```

Возможные причины:

```text
1. запущено несколько экземпляров бота
2. старый процесс не остановлен
3. Docker-контейнер и локальный python работают одновременно
4. один handler ловит слишком широкий набор событий
5. callback payload совпадает у разных разделов
6. нет return после обработки события
7. middleware повторно прокидывает событие
8. MAX прислал retry, а бот не делает idempotency
```

### Проверка нескольких процессов

Локально на Windows:

```powershell
Get-Process | Where-Object { $_.ProcessName -like "*python*" }
```

На Linux-сервере:

```bash
ps aux | grep python
docker compose ps
docker ps
```

Если одновременно работают:

```text
локальный бот
серверный бот
старый контейнер
новый контейнер
```

MAX может доставлять события в несколько активных polling-процессов. Это дает дубли.

Правило:

```text
в production должен работать один экземпляр polling-бота
```

### Точная фильтрация handlers

Плохо:

```python
@router.message()
async def any_message(event):
    await ask_question(event)
```

Такой handler ловит все:

```text
/start
/balance
админ-команды
обычные ответы
технические сообщения
```

Хорошо:

```python
@router.message()
async def waiting_question(event, state):
    if await state.get() != "waiting_question":
        return
    ...
```

Для callback:

```python
if payload != "tarot:ask":
    return
```

### Namespace для callback payload

Все payload должны иметь префиксы:

```text
menu:main
tarot:ask
tarot:love
tarot:money
balance:show
pay:week
pay:month
admin:stats
admin:help
mailer:ask
```

Нельзя:

```text
start
back
stats
yes
no
```

Потому что такие payload быстро начинают конфликтовать.

### Idempotency

Если MAX присылает один и тот же event повторно, бот не должен выполнять платеж или генерацию второй раз.

Для важных действий нужно использовать:

```text
event_id dedupe
user lock
payment lock
generation lock
transaction status check
pending_action is_active check
```

Пример:

```text
если transaction уже paid:
  не активировать подписку повторно
```

```text
если pending_action уже is_active=0:
  не запускать генерацию повторно
```

### Ошибки, которые нельзя скрывать

Нельзя молча проглатывать:

```text
ошибки платежей
ошибки списания баланса
ошибки создания пользователя
ошибки генерации
ошибки БД
ошибки миграций схемы
```

Можно тихо игнорировать:

```text
не удалось удалить progress-сообщение
не удалось отправить админу, который заблокировал бота
не удалось отредактировать старое сообщение, если уже отправлен новый экран
```

Даже тихие ошибки лучше писать в debug log.

### Пользовательские ошибки

Пользователь не должен видеть traceback.

Плохо:

```text
HandlerException: NameError: crud is not defined
```

Хорошо:

```html
Не получилось выполнить действие.
Попробуйте еще раз или вернитесь в главное меню.
```

Кнопка:

```text
🏠 Главное меню
```

Админу при этом отправить:

```html
❌ <b>Ошибка обработчика</b>
Router: <code>tarot</code>
User: <code>5462936</code>
Ошибка: <code>NameError: crud is not defined</code>
```

### Логи платежей

Для каждого платежа:

```text
created
pending
succeeded
canceled
expired
paid
```

Лог должен содержать:

```text
user_id
transaction_id
provider_payment_id
plan_id
amount
status
```

Нельзя логировать полный секретный payload YooKassa.

### Логи генерации

Для каждой генерации:

```text
user_id
scenario
cost
prompt length
started_at
finished_at
duration
success/fail
```

Если генерация упала:

```text
refund done: yes/no
```

### Чек-лист этапа

```text
[ ] логи содержат user_id и chat_id
[ ] ошибки handlers логируются с router_id
[ ] callback payload имеют namespace
[ ] широкие handlers ограничены state/filter
[ ] есть lock для генерации
[ ] есть lock для платежей
[ ] transaction paid не обрабатывается дважды
[ ] pending_action не запускается дважды
[ ] пользователь не видит traceback
[ ] админ получает короткое техническое уведомление об ошибке
[ ] проверено, что запущен только один экземпляр бота
```

### Результат этапа

В план миграции агент записывает:

```text
Этап 12 завершен.
Логирование приведено к MAX-событиям.
Дубли обработчиков проверены.
Callback payload разделены namespace.
Добавлены lock-и генерации и платежей.
Пользовательские ошибки отделены от технических логов.
```

## 13. Этап Docker, сервера, внешней БД и деплоя

### Цель этапа

Подготовить MAX-бота к запуску на сервере так, чтобы:

1. код можно было обновлять без потери БД;
2. база лежала вне контейнера;
3. временные файлы лежали вне контейнера;
4. `.env` не попадал в git;
5. контейнер автоматически перезапускался;
6. деплой можно было повторить понятными командами.

### Файлы деплоя

В проекте должны быть:

```text
Dockerfile
docker-compose.yml
.dockerignore
.env.example
README.md
```

`.env` должен быть на сервере, но не должен быть в git.

`.env.example` должен быть в git, но без секретов.

### Dockerfile

Базовый принцип:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["python", "main.py"]
```

Если проект использует медиа/ffmpeg:

```dockerfile
RUN apt-get update \
    && apt-get install -y --no-install-recommends ffmpeg \
    && rm -rf /var/lib/apt/lists/*
```

### docker-compose.yml

Главное правило:

```text
БД и media должны быть volume снаружи контейнера.
```

Пример:

```yaml
services:
  bot:
    build: .
    container_name: maxtarobot
    env_file:
      - .env
    volumes:
      - ./data:/app/data
      - ./media:/app/media
    restart: unless-stopped
```

В `.env`:

```env
DATABASE_PATH=/app/data/database.db
MEDIA_TEMP_DIR=/app/media/temp
```

На сервере:

```text
/root/maxtarobot/data/database.db
/root/maxtarobot/media/temp/
```

### .dockerignore

Нужно исключить:

```text
.git
__pycache__
*.pyc
.env
data
media/temp
.venv
venv
old_telegram_bot
previous_telegram_bot
```

Если нужно деплоить стартовую БД, это отдельное осознанное действие. По умолчанию БД на сервер не перетирается.

### .gitignore

Обязательно:

```text
.env
data/
media/temp/
__pycache__/
*.pyc
.venv/
venv/
```

Если в проекте был backup Telegram-бота, его нужно либо:

```text
не добавлять в git
```

либо:

```text
удалить после завершения миграции
```

В финальном MAX-репозитории не должно быть старой папки Telegram-бота, если она уже не нужна.

### Серверная структура

Рекомендуемая структура:

```text
/root/maxtarobot/
  main.py
  config.py
  requirements.txt
  docker-compose.yml
  Dockerfile
  .env
  data/
    database.db
  media/
    temp/
  handlers/
  max_handlers/
  services/
  database/
  keyboards/
  prompts/
```

### Безопасная выгрузка проекта

Перед выгрузкой:

```text
1. проверить git status
2. проверить, что .env не попадет в архив/git
3. проверить, что data/database.db не будет случайно перетерта
4. сделать backup БД на сервере
```

Backup БД:

```bash
mkdir -p /root/maxtarobot/backups
cp /root/maxtarobot/data/database.db /root/maxtarobot/backups/database_$(date +%Y%m%d_%H%M%S).db
```

### Команды деплоя на сервере

Перейти в папку:

```bash
cd /root/maxtarobot
```

Собрать и запустить:

```bash
docker compose up -d --build
```

Посмотреть статус:

```bash
docker compose ps
```

Логи:

```bash
docker compose logs -f bot
```

Перезапуск:

```bash
docker compose restart bot
```

Остановка:

```bash
docker compose down
```

### Проверка, что БД примонтирована снаружи

На сервере:

```bash
ls -lah /root/maxtarobot/data
```

В контейнере:

```bash
docker compose exec bot ls -lah /app/data
```

Путь в логах должен совпадать с:

```text
DATABASE_PATH=/app/data/database.db
```

Если бот создал БД внутри контейнера не в `/app/data`, значит volume настроен неправильно.

### Проверка, что работает один бот

На сервере:

```bash
docker compose ps
docker ps
ps aux | grep python
```

Не должно быть:

```text
старого python main.py вне Docker
старого контейнера с тем же ботом
локального polling, который тоже подключен к этому token
```

Если пользователь получает дубли сообщений, это первая проверка.

### Обновление `.env` на сервере

Правило:

```text
локальный .env не пушить в git
серверный .env обновлять вручную или через безопасную передачу
после изменения .env перезапускать контейнер
```

Команды:

```bash
nano /root/maxtarobot/.env
docker compose restart bot
docker compose logs -f bot
```

### Миграции БД

При старте бот должен сам выполнять:

```text
setup_db
ensure_schema
ALTER TABLE IF NEEDED
CREATE INDEX IF NOT EXISTS
```

Нельзя вручную удалять production БД ради новой схемы.

Если нужна ручная миграция:

```text
1. сделать backup
2. проверить SQL локально
3. выполнить на сервере
4. проверить логи старта
```

### Чек-лист этапа

```text
[ ] Dockerfile есть
[ ] docker-compose.yml есть
[ ] .dockerignore исключает .env, data, media/temp
[ ] .gitignore исключает .env и БД
[ ] DATABASE_PATH указывает на /app/data/database.db
[ ] ./data примонтирована в контейнер
[ ] перед деплоем сделан backup production БД
[ ] docker compose up -d --build проходит без ошибок
[ ] docker compose logs не показывает traceback
[ ] работает только один экземпляр бота
[ ] после restart БД сохраняется
```

### Результат этапа

В план миграции агент записывает:

```text
Этап 13 завершен.
Docker-конфигурация подготовлена.
БД вынесена в ./data и примонтирована в /app/data.
Media вынесены в ./media.
.env не попадает в git.
Проект запущен через docker compose.
Проверено, что работает один экземпляр бота.
```

## 14. Финальный smoke-test

### Цель этапа

Проверить не код по отдельности, а живой пользовательский путь.

Smoke-test должен отвечать на вопрос:

```text
можно ли обычному пользователю открыть бота, пройти сценарий, оплатить и получить результат?
```

### Подготовка к тесту

Перед тестом:

```text
1. запустить бота
2. открыть логи
3. проверить .env
4. проверить DATABASE_PATH
5. проверить ADMIN_IDS и ADMIN_NOTIFY_IDS
6. проверить, что админ уже писал боту
7. проверить, что нет второго процесса бота
```

Логи держать открытыми:

```bash
docker compose logs -f bot
```

### Тест 1. Старт

Действия:

```text
1. открыть бота в MAX
2. отправить /start
3. нажать Главное меню
```

Ожидаемо:

```text
бот отвечает одним приветствием
появляется inline-меню
в БД появился пользователь
админ получил уведомление о новом пользователе
повторный /start не создает дубль
```

### Тест 2. Главное меню

Действия:

```text
нажать каждую основную кнопку меню
```

Ожидаемо:

```text
каждая кнопка дает понятный экран
нет нескольких разных ответов на одну кнопку
есть возврат в главное меню
нет traceback в логах
```

### Тест 3. Сценарий генерации без оплаты

Если у пользователя есть баланс:

```text
1. нажать целевую кнопку
2. ввести вопрос или prompt
3. дождаться результата
```

Ожидаемо:

```text
бот отправляет стикер или progress
бот отправляет одно process-сообщение
после результата progress удаляется, если API позволяет
баланс списывается один раз
результат приходит корректно
админ получает success-log
```

### Тест 4. Недостаточно баланса

Действия:

```text
1. установить пользователю balance=0
2. начать генерацию
3. ввести вопрос
```

Ожидаемо:

```text
бот не теряет вопрос
создается pending_action
показывается экран оплаты
```

### Тест 5. Оплата

Действия:

```text
1. выбрать тариф
2. перейти по ссылке YooKassa
3. завершить оплату
4. дождаться polling
```

Ожидаемо:

```text
transaction -> paid
subscription -> active
balance начислен
пользователь получил "Оплата прошла успешно"
если был pending_action, сценарий продолжился
админ получил payment-log
```

### Тест 6. Баланс и подписка

Действия:

```text
1. открыть /balance
2. нажать обновить подписку
3. нажать отключить подписку
```

Ожидаемо:

```text
карточка подписки показывает правильный тариф
остаток раскладов правильный
дата обновления правильная
отключение не ломает экран
админ получает уведомление об отключении
```

### Тест 7. Админ-команды

Проверить:

```text
/admin_help
/botstats
/sub_check <ID>
/sub_on <ID> 5
/sub_off <ID>
/genpromo 5
/admin_list
```

Ожидаемо:

```text
админ получает ответы
обычный пользователь не может выполнить команды
команды в /admin_help удобно копировать
```

### Тест 8. Deeplinks

Проверить:

```text
ref_<id>
promo_<code>
utm tag
```

Ожидаемо:

```text
referrer_id сохраняется
utm_source сохраняется
promo применяется
повторный start не перетирает старые значения
самореферал игнорируется
```

### Тест 9. Рассылка

Действия:

```text
1. временно сократить интервал рассылки в test-режиме
2. дождаться preview админам
3. дождаться старта
4. проверить push у тестового пользователя без подписки
5. нажать кнопку "Задать вопрос"
```

Ожидаемо:

```text
preview пришел админам
start пришел админам
push содержит inline-кнопку
кнопка ведет в сценарий вопроса
finish содержит статистику
пользователи с активной подпиской не получают push, если аудитория настроена "без активной подписки"
```

### Тест 10. Перезапуск контейнера

Действия:

```bash
docker compose restart bot
```

Ожидаемо:

```text
бот стартует без ошибок
БД сохранилась
подписки сохранились
балансы сохранились
рассылка не запустилась дважды
```

## 15. Готовый prompt для LLM-агента

### Как использовать этот раздел

Этот prompt можно дать другой нейросети вместе с кодом Telegram-бота.

Лучший формат:

```text
1. приложить исходный проект Telegram-бота
2. приложить этот файл TELEGRAM_TO_MAX_AGENT_GUIDE.md
3. приложить текущий agents.md проекта, если он есть
4. дать prompt ниже
5. попросить нейросеть работать по этапам и фиксировать прогресс в MAX_MIGRATION_PLAN.md
```

### Основной prompt

```text
Ты выступаешь как senior Python-разработчик и миграционный агент.

У тебя есть готовый рабочий Telegram-бот. Нужно перенести его в мессенджер MAX.

Главная цель: не переписать проект с нуля, а сохранить бизнес-логику и заменить Telegram-слой на MAX-слой.

Сохраняем:
- БД и CRUD-логику;
- пользователей;
- баланс;
- подписки;
- YooKassa;
- pending_actions;
- deeplink-атрибуцию;
- промокоды;
- админку;
- рассылки;
- фоновые задачи;
- внешние API генерации;
- бизнес-экономику тарифов.

Заменяем:
- Telegram handlers;
- Telegram keyboards;
- Telegram commands;
- Telegram-specific payments вроде Stars, если MAX их не поддерживает;
- отправку сообщений;
- callback обработку;
- формат deeplink под MAX;
- UX-тексты под интерфейс MAX;
- Docker/env настройки, если они завязаны на Telegram.

Работай строго по файлу TELEGRAM_TO_MAX_AGENT_GUIDE.md.

Сначала сделай аудит проекта:
1. найди точку входа;
2. найди Telegram-зависимости;
3. найди все handlers;
4. найди платежи;
5. найди подписки;
6. найди генерацию;
7. найди рассылку;
8. найди админку;
9. найди схему БД;
10. найди .env переменные.

Создай файл MAX_MIGRATION_PLAN.md.
В нем фиксируй каждый этап:
- что найдено;
- что перенесено;
- какие файлы изменены;
- что проверено;
- какие риски остались.

Перед началом полноценной миграции сделай echo-тест MAX:
1. создай минимальный тестовый скрипт;
2. подключи MAX_BOT_TOKEN;
3. проверь, что бот получает сообщение;
4. проверь, что бот отвечает;
5. зафиксируй результат в MAX_MIGRATION_PLAN.md.

После успешного echo-теста переноси проект этапами:

Этап 1. Аудит исходного Telegram-бота.
Этап 2. Подготовка структуры MAX-проекта.
Этап 3. Проверка MAX API.
Этап 4. Создание MAX entrypoint.
Этап 5. Интерфейс и дизайн MAX-бота.
Этап 6. /start, deeplink, ref, promo, UTM.
Этап 7. Продуктовый сценарий генерации.
Этап 8. YooKassa и pending_actions.
Этап 9. Подписки и subscription_watcher.
Этап 10. Админка, команды, права, статистика.
Этап 11. Smart-рассылка и push-шаблоны.
Этап 12. Логи, ошибки, диагностика дублей.
Этап 13. Docker, сервер, внешняя БД и деплой.
Этап 14. Финальный smoke-test.

Правила интерфейса:
- все меню делать через inline-кнопки;
- не отправлять много сообщений подряд, если можно одним;
- каждый экран должен иметь понятный следующий шаг;
- во вторичных разделах должна быть кнопка "Главное меню";
- callback payload должны иметь namespace: menu:, tarot:, pay:, admin:, mailer:;
- команды в админской помощи писать в <code>, чтобы их было удобно копировать;
- тексты делать короткими, понятными и без технических терминов;
- progress генерации делать одним стикером и одним текстовым сообщением;
- после результата удалять progress, если MAX API позволяет;
- если удалить нельзя, бот не должен падать.

Правила платежей:
- YooKassa остается основным платежным методом;
- Telegram Stars не переносить в MAX, если MAX явно не поддерживает аналог;
- при успешной оплате сначала отправить пользователю сообщение "Оплата прошла успешно";
- если есть pending_action, после сообщения об оплате продолжить сценарий;
- повторная обработка paid transaction запрещена;
- pending_action нельзя запускать дважды.

Правила подписок:
- активная подписка определяется по status и current_period_end;
- auto_renew отвечает только за автопродление;
- отключение подписки не обнуляет баланс до конца периода;
- при окончании периода без auto_renew баланс обнуляется;
- при продлении старый остаток сгорает и заменяется новым лимитом тарифа.

Правила рассылки:
- по умолчанию рассылать пользователям без активной подписки;
- периодичность 12 часов;
- за 30 минут отправлять preview админам;
- на старте уведомлять админов;
- после завершения отправлять статистику;
- push-тексты хранить в отдельном файле и крутить по кругу;
- не вызывать нейросеть для каждого push, если есть готовые шаблоны;
- к push добавлять inline-кнопку целевого действия.

Правила деплоя:
- .env не добавлять в git;
- БД хранить вне контейнера в ./data;
- media/temp хранить вне контейнера в ./media;
- перед деплоем делать backup production БД;
- запускать через docker compose;
- убедиться, что работает только один экземпляр бота.

После каждого крупного этапа:
1. обнови MAX_MIGRATION_PLAN.md;
2. коротко напиши, что сделано;
3. запусти доступные проверки;
4. не переходи к деплою, пока локальный запуск не работает.

В финале выдай отчет:
1. что перенесено;
2. какие файлы изменены;
3. какие сценарии проверены;
4. что не проверено;
5. какие риски остались;
6. какие команды нужны для запуска;
7. как обновлять проект на сервере.
```

## Финальное правило для агента

Если есть сомнение, что менять:

```text
сначала сохранить рабочую бизнес-логику,
потом заменить Telegram-интерфейс,
потом проверить живой пользовательский путь.
```

Нельзя одновременно:

```text
менять БД,
переписывать платежи,
переписывать генерацию,
менять интерфейс,
и деплоить на сервер.
```

Правильный порядок:

```text
аудит -> echo MAX -> интерфейс -> сценарии -> платежи -> фоновые задачи -> тесты -> деплой
```
